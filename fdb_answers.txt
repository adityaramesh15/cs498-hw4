**Q1:** FDB decouples the Transaction Management System and Storage System so they can scale independently based on the workload. The TMS handles the quick, memory-intensive work of processing transactions and checking for conflicts. Meanwhile, the SS manages the actual disk storage and data replication, which is I/O bound. This separation prevents slow disk writes from bottlenecking our fast transaction processing.

**Q2:** The Resolvers within the Transaction Management System are responsible for detecting read-write conflicts between concurrent transactions. This check happens right at the beginning of the commit phase, before any changes are actually written to disk. If the Resolver sees that someone else modified the data I read during my transaction, it aborts my commit so I can safely retry it.

**Q3:** The Sequencer gives my transaction a read version to ensure I see a perfectly consistent, point-in-time snapshot of the database. This guarantees strict serializability by making sure my reads don't capture partial updates from other transactions happening simultaneously. The commit version is then used to timestamp my successful writes, moving the database state forward for the next set of transactions.

**Q4:** Unlike standard reads, snapshot reads completely bypass the conflict detection process, meaning my transaction won't be aborted if someone else modifies the same data. I would choose a snapshot read when running broad analytics or exporting data where perfect, up-to-the-millisecond consistency isn't strictly necessary. This helps avoid unnecessary transaction failures on long-running read operations.

**Q5:** FDB achieves "cheap" recovery because the LogServers actively push transaction logs to the StorageServers during normal operation, rather than waiting for a crash to figure things out. When a crash happens, the StorageServers already have most of the data they need, so the system just has to agree on the final commit version. This means we don't have to waste time heavily replaying a massive redo log before the database can accept new queries.
